# 关键决定记录

## D-001：保留现有 React + Vite + NestJS

决定：不新建 Next.js，不推倒现有 Talent MVP。

原因：当前已有可运行闭环，平台化应渐进迁移，避免浪费代码和额度。

## D-002：先做平台底座，不先接完整飞书

决定：第一阶段先做 Task Runtime + Model Gateway。

原因：API 稳定性与可追踪性是当前最大痛点，飞书必须接入统一任务链，而不是直接连接某个业务服务。

## D-003：飞书是入口，不是核心数据库

决定：飞书用于身份、消息、卡片、审核、协作视图；平台保存主数据。

原因：工作流节点、模型日志、文件关系和权限不适合全部依赖多维表格。

## D-004：Agent 绑定 Model Profile

决定：业务 Agent 不直接绑定 Provider 和模型名。

原因：便于切换、回退、成本控制和能力路由。

## D-005：第一版不做拖拽工作流

决定：Workflow 先用代码或配置定义。

原因：拖拽编辑器开发量大，但不会直接解决模型稳定性和业务验证问题。

## D-006：先用 JSON Repository，再迁移 PostgreSQL

决定：阶段 1 保留本地 JSON，但必须通过 Repository 接口。

原因：降低第一阶段改造范围，同时为生产数据库留出替换边界。

## D-007：第二个 Agent 选择 Prompt 质检

决定：完成平台底座后优先接 Prompt 质检 Agent。

原因：输入简单、规则明确、容易验收，适合验证平台复用能力。

## D-008：两台电脑、两个 Codex 账号以 Git 为唯一接力来源

决定：不依赖聊天历史，使用 `PROJECT_MEMORY.md`、`STATUS.md` 和 Git 历史。

原因：两个账号的对话、记忆和额度互不共享。
